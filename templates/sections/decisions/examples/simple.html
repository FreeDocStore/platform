ADR-1. Run on Cloudflare Workers rather than a long-lived service

Status: Accepted

Date: 2026-04-08

Context

The tool handles fewer than 50 requests per day, must respond inside Slack's 3-second window, and is bound by an explicit "no new vendors" mandate. The team already runs other Workers in production and has the operational pattern internalised.

Decision

Deploy the tool as a single Cloudflare Worker. The Worker handles the slash command directly with no auxiliary services.

Consequences

Easier:

  • Zero new infrastructure - reuses the team's existing Cloudflare account, Wrangler workflow, and on-call playbook.
  • Cold-start latency is irrelevant on Workers; Slack's 3-second budget is comfortable.
  • Cost is effectively zero at this volume (well within the free tier).

Harder:

  • Postgres connectivity from Workers requires a serverless-compatible driver; we use @neondatabase/serverless, which is one more dependency to track.
  • If the tool ever needs to do background work (e.g., bulk audit replays), Workers' execution-time limits will force a refactor to a different runtime.

Alternatives considered

  • A new Node service on the existing app-services platform - rejected on the no-new-services constraint and on overkill for the volume.
  • Embedding the lookup in the existing internal admin tool - rejected because the admin tool is not exposed to Slack and adding Slack auth there is more work than this entire project.

ADR-2. Read Stripe live on every request, never cache

Status: Accepted

Date: 2026-04-08

Context

Subscription state can change at any time (upgrades, downgrades, cancellations, dunning). Stale data shown to a support agent is worse than slightly slower data - agents quote what they see, and quoting an out-of-date plan creates customer-service incidents. Stripe's read API is fast enough to satisfy the response budget.

Decision

Every slash command does a fresh read against Stripe's /customers and /subscriptions endpoints. No in-Worker cache, no shared cache, no read-through layer.

Consequences

Easier:

  • No cache invalidation logic - the source of truth is always Stripe.
  • Agents can never quote stale plan data.
  • Audit log is exact: each lookup corresponds 1:1 with a real Stripe call.

Harder:

  • If usage grows beyond ~1 req/sec sustained, we will hit Stripe's per-account read rate limits and need to introduce caching - which means revisiting this decision.
  • Latency is bounded by Stripe's API performance, not by anything we control.

Alternatives considered

  • Cache plan name and renewal date for 5 minutes in a Cloudflare KV namespace - rejected because plan changes within 5 minutes are exactly the moments support agents need accurate data (e.g., a customer just upgraded and is calling about the upgrade).
  • Mirror Stripe webhook events into our own datastore - rejected as more infrastructure than the problem warrants. Reasonable to revisit if volume grows.

ADR-3. Audit-write failures do not fail the lookup

Status: Accepted

Date: 2026-04-09

Context

The Worker writes an audit row to the company's internal Postgres after each successful lookup. The audit table is hosted on a shared internal database that has had short outages historically (~quarterly, lasting minutes). Security requires audit logging; Support requires the lookup to work even when the audit DB is down.

Decision

The audit-write happens on a fire-and-forget code path. If the write fails, the Worker emits a metric and returns the lookup result anyway. A daily reconciliation job (existing internal tooling) catches missing audit rows and back-fills from the metric stream.

Consequences

Easier:

  • Lookup availability is decoupled from audit-DB availability.
  • Agents are unblocked during DB incidents.

Harder:

  • Audit completeness depends on the daily reconciliation job, which adds a small operational responsibility.
  • If the reconciliation job ever lags more than 24 hours, audit gaps are possible. The existing internal tooling handles this with paging on lag, but it adds a moving part.

Alternatives considered

  • Block the lookup on a successful audit write (synchronous) - rejected because every shared-DB outage would manifest as a Support tool outage.
  • Buffer audit rows in Cloudflare KV and flush periodically - rejected because Workers KV's eventual-consistency window can lose writes within minutes; not acceptable for an audit log.
