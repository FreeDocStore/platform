ADR-1. Single-tenant per agency, never multi-tenant

Status: Accepted

Date: 2026-04-12

Context

The agency's data-residency rules and audit obligations are agency-specific. The state oversight body audits each agency separately. A multi-tenant deployment would entangle audit boundaries and complicate residency proofs. Future agencies might want similar platforms but their compliance frame is not guaranteed identical.

Decision

The platform is deployed as a single-tenant instance per agency. Database, blob storage, and identity integration are scoped to one agency. There is no shared layer that crosses agencies.

Consequences

Easier:

  • Compliance evidence is per-instance and unambiguous - one Postgres, one blob account, one audit log workspace.
  • Data residency is enforced at the resource level, not the application level.
  • Per-agency customisation (e.g., risk tiers, outcome codes) does not require a tenant-scoped configuration system.

Harder:

  • Operating multiple agencies later means multiple deployments to maintain - more environments, more upgrade work.
  • No shared analytics across agencies (which the agency leadership would not allow anyway, but worth naming).
  • A future product line for the delivery partner cannot trivially share infrastructure across customers.

Alternatives considered

  • Multi-tenant with row-level security - rejected because the audit and compliance boundary lives outside the application; row-level security is application-enforced and harder to defend in audit.
  • Multi-tenant with separate databases per tenant in the same cluster - rejected for the same reason; the cluster itself becomes a shared resource that complicates residency claims.

ADR-2. PostgreSQL primary, no document database

Status: Accepted

Date: 2026-04-13

Context

Case data is highly relational (cases reference clients, who reference households, who reference services, etc.). Reporting requirements demand complex joins. Compliance demands transactional consistency for state changes. The agency's platform team supports PostgreSQL as a managed service in Azure; they do not operate any document database in production.

Decision

PostgreSQL 16 is the only operational database. All case data, audit references, and reconciliation state live there. Free-text and document-like data (e.g., long case notes) are stored as JSONB columns within the same database.

Consequences

Easier:

  • One database to operate, back up, and restore. One residency boundary.
  • Reporting can use SQL joins directly; no cross-store joins to manage.
  • Transactional integrity is straightforward (case state changes + audit references in one transaction).

Harder:

  • Free-text search on case notes is more limited than a dedicated search engine would offer; will use Postgres full-text search initially and revisit if performance suffers.
  • If we ever need true document-database scale (millions of documents per case), we will need to introduce object storage with pointers - planned for blobs already, but not for searchable text.

Alternatives considered

  • Postgres + Cosmos DB for case notes - rejected because the platform team would need to learn a second operational surface, and the volume does not justify it.
  • Postgres + Elasticsearch for search - deferred. Postgres FTS is the v1 choice; Elasticsearch is on the v2 roadmap if FTS performance fails the requirements.

ADR-3. Server-rendered with Remix, not a SPA

Status: Accepted

Date: 2026-04-15

Context

Caseworkers use the platform on agency-supplied tablets over variable cellular connectivity. Page-load performance on slow networks is a stated NFR (under 800 ms p95 on 4G). SPAs ship more JavaScript per session and rely on a fast initial connection to download the bundle. Server-rendering reduces the initial payload and renders meaningful content sooner.

Decision

The web tier is built with Remix, which renders HTML on the server and progressively hydrates. Forms work without JavaScript; client-side enhancement is layered on for richer interactions but is not required for core flows.

Consequences

Easier:

  • First-paint performance is consistent across network conditions.
  • Forms work even if hydration fails - critical for caseworkers in rural cellular dead-zones.
  • SEO and accessibility are easier; standard HTML semantics work out of the box.

Harder:

  • Real-time features (live caseload counters for supervisors) require explicit choices: WebSockets via Remix's resource routes, or polling. Not "free" the way it is in a fully client-driven SPA.
  • Some interaction patterns (drag-and-drop reassignment) need richer client logic, which is more work in a server-first architecture than in an SPA.

Alternatives considered

  • Next.js - rejected on team familiarity; the delivery team has shipped four Remix apps and zero Next apps. Both are credible options.
  • Single-page React app with TanStack Router - rejected on the page-load NFR; the bundle size required for the case workspace exceeded the 4G budget in early prototyping.
  • Server-rendered Django - rejected on language alignment; the rest of the platform is TypeScript and the team prefers a single language.

ADR-4. Australia East only, no multi-region failover

Status: Accepted

Date: 2026-04-16

Context

The state's data-residency rules require that all data at rest and in transit remain within the country. Most multi-region patterns implicitly replicate to a second region for failover; doing so to a region outside the state would violate residency. The two Australian Azure regions (Australia East and Australia Southeast) are both compliant, but operating cross-region active-passive doubles infrastructure cost and complicates the audit story.

Decision

The platform deploys to Australia East only. Disaster recovery uses Azure's geo-redundant storage within Australia (snapshots replicated to Australia Southeast in cold storage). RTO is 4 hours; RPO is 15 minutes via Postgres point-in-time recovery.

Consequences

Easier:

  • Single active region simplifies audit and residency claims significantly.
  • Lower operational cost - one production environment, not two.
  • No application-level changes needed for cross-region consistency.

Harder:

  • An Australia East regional outage takes the platform down for the duration of the outage. Caseworkers cannot work; supervisors cannot reassign. Business continuity falls back to the legacy paper process.
  • Recovery from a regional disaster takes hours (RTO 4h), not minutes. This is documented in the runbook as an accepted risk.

Alternatives considered

  • Active-passive across Australia East and Australia Southeast - rejected on cost and complexity; the agency leadership accepts the regional outage risk in exchange for a leaner operating model.
  • Active-active multi-region - not viable; PostgreSQL primary cannot be active in two regions simultaneously without significant write-conflict tooling, which is overkill at this scale.

ADR-5. Audit log goes to Log Analytics, not the operational database

Status: Accepted

Date: 2026-04-18

Context

The agency must retain audit records for 7 years to satisfy state safeguarding rules. Storing 7 years of audit rows in the operational Postgres database degrades performance over time and complicates backups. The audit data is append-only, queried infrequently, and retrieved primarily by auditors. Azure Log Analytics handles long-retention append-only data well and is already approved by the agency security team.

Decision

The api emits an audit event for every record-mutating action. Events are written to a dedicated Azure Log Analytics workspace with 7-year retention. The operational Postgres holds a 30-day rolling cache of audit references for in-app display; the canonical audit record lives in Log Analytics.

Consequences

Easier:

  • Operational Postgres stays small and fast indefinitely.
  • Long-retention compliance is solved by Log Analytics' built-in retention features.
  • Audit queries (KQL) are the right tool for the job - faster than equivalent SQL on a 7-year-old table.

Harder:

  • Two stores to manage. The 30-day cache and the canonical Log Analytics record can drift if the emit-then-cache pattern fails partway. Mitigated by a daily reconciliation job that flags drift.
  • Auditors must learn KQL to query the canonical record. The platform team will provide canned queries for the common audit shapes.

Alternatives considered

  • Audit rows in the operational Postgres with an archival job to blob storage - rejected because querying archived audit rows requires un-archival; auditors would need engineering support for every query.
  • Dedicated Postgres instance for audit data - rejected as another database to operate when Log Analytics covers the use case.
