Architecture

A multi-component Azure-hosted case management platform: web app, API service, Postgres primary, blob-stored attachments, integrated with the agency SSO and read-only finance API.

Summary

Caseworkers and supervisors interact with a server-rendered web app delivered from Azure App Service in the agency's mandated region. The web app calls a private API service that owns all writes to the Postgres primary and to Azure Blob Storage for case attachments. Authentication terminates at the SSO provider via SAML; sessions are issued as signed cookies by the API. A nightly job reconciles billing line items against the agency's existing finance system over a read-only API. The platform is single-tenant by design; no other agencies share the deployment.

Components

web

  • Purpose: server-rendered HTML for caseworkers and supervisors. Owns intake forms, case workspaces, supervisor dashboards.
  • Language / framework: TypeScript, Remix on Node 20.
  • Depends on: api, SSO provider.
  • Owner: delivery team.

api

  • Purpose: the only writer to the database and blob store. Owns authorisation, validation, audit-log emission, finance reconciliation.
  • Language / framework: TypeScript, Node 20, Fastify, Prisma.
  • Depends on: Postgres, Blob Storage, finance API (read-only), audit log sink.
  • Owner: delivery team.

db (Postgres primary)

  • Purpose: single source of truth for case data. One read replica for reporting workload.
  • Language / framework: PostgreSQL 16, Azure Database for PostgreSQL Flexible Server.
  • Depends on: none.
  • Owner: delivery team; agency platform team owns underlying Azure resource.

blob

  • Purpose: case attachments (PDFs, images, audio notes). References live in Postgres; blobs live here.
  • Language / framework: Azure Blob Storage, hot tier, locally-redundant.
  • Depends on: none.
  • Owner: agency platform team.

finance-sync

  • Purpose: nightly batch job. Pulls billing items from the finance API, reconciles against case records, writes a daily report.
  • Language / framework: TypeScript, runs as an Azure Container App scheduled job.
  • Depends on: api (write reconciliation results), finance API (read).
  • Owner: delivery team.

Data flow

Case intake submission

  1. Caseworker completes the intake form in web and submits.
  2. Web POSTs the validated payload to api with the user's signed session cookie.
  3. API authorises the request against the user's role and the case ownership rules.
  4. API writes the case row plus risk-assessment row in a single Postgres transaction.
  5. If attachments were uploaded, web has already streamed them directly to blob via SAS URLs; api stores the references.
  6. API emits an audit-log entry: user ID, action case.intake, case ID, before/after state.
  7. API returns the case ID; web redirects the caseworker to the new case workspace.

Supervisor reassignment

  1. Supervisor invokes "Reassign" on a case in the supervisor dashboard.
  2. Web POSTs the reassignment (target user, reason) to api.
  3. API authorises (only supervisors with the case in their region can reassign).
  4. API updates the case row, emits two audit entries (one for the reassignment event, one for the case-state change).
  5. API enqueues notifications to the previous and new caseworker via the platform's email service.

External dependencies

  • Agency SSO (SAML 2.0) - identity provider for all users. When down: no new logins; existing sessions continue until expiry. Documented as the single auth dependency.
  • Agency finance API (read-only) - reconciliation source. When down: nightly sync skips and alerts; manual reconciliation possible the next day.
  • Email service (agency-managed Office 365) - notifications. When down: notifications queue; case workflow continues without them.
  • Azure Blob Storage - case attachments. When down: existing case data still accessible via api/db; new uploads blocked. Read-only mode for the workspace.

Tech stack

  • Language: TypeScript 5.x across web, api, finance-sync
  • Runtime: Node 20 (web, api), Azure Container Apps (finance-sync), Azure App Service (web, api hosting)
  • Framework: Remix (web), Fastify (api), Prisma (data layer)
  • Database: PostgreSQL 16 (Azure Database for PostgreSQL Flexible Server) with one read replica
  • Storage: Azure Blob Storage (hot tier, locally-redundant)
  • Key libraries: @prisma/client, fastify, fastify-jwt, @azure/storage-blob, react-hook-form, zod (validation)

Deployment topology

  • Cloud account / region: agency Azure tenancy, Australia East (mandated region for state data residency).
  • Environments: dev, staging, production - separate resource groups, separate databases.
  • Ingress: Azure Front Door fronts the production web tier; TLS 1.3 terminates at Front Door; backend traffic is private (VNet-integrated).
  • Data residency: all Postgres rows, blob objects, and backups remain in Australia East. Cross-region replication is explicitly disabled. Disaster recovery is geo-redundant within the same country.

Cross-cutting concerns

Security

Auth terminates at the agency SSO via SAML 2.0 (api as service provider). Sessions are signed JWTs in HttpOnly SameSite=Lax cookies. Supervisor and admin roles require MFA enforced at the IdP. All inter-service traffic is mTLS-authenticated within the VNet. Detailed security model in Security model.

Availability

Production target 99.9% measured monthly, excluding declared maintenance windows. RPO 15 minutes (Postgres point-in-time recovery), RTO 4 hours. Failure-mode handling and runbooks live in Runbook.

Observability

Application metrics, logs, traces emit to Azure Monitor (Application Insights). Dashboards in the Azure portal; alerts route to PagerDuty for the on-call rotation. Audit log emits to a separate Azure Log Analytics workspace with 7-year retention to satisfy state safeguarding rules.

Scale

Sized for 200 concurrent users and 10x peak headroom. Web tier scales horizontally on Azure App Service (autoscale rules on CPU and request queue depth). API tier scales identically. Postgres primary is single-node (no horizontal scale needed at this volume); read replica handles reporting load. Identified bottleneck under future growth: blob upload concurrency for evidence-heavy cases.
