Architecture

A single Cloudflare Worker that exposes a Slack slash command, calls Stripe's API, and writes audit rows to an existing Postgres database.

Summary

The tool is a stateless edge function with no datastore of its own. Slack POSTs slash-command payloads to a Cloudflare Worker, which validates the signature, queries Stripe for customer subscription state, formats the response as Slack message blocks, and returns it. An audit row is written asynchronously to the company's existing Postgres audit table via a connection string in the Worker's environment.

Components

sub-status worker

  • Purpose: handles all Slack slash-command requests; runs the lookup, formats the response, writes audit row.
  • Language / framework: TypeScript, Cloudflare Workers runtime.
  • Depends on: Stripe API, Postgres audit table.
  • Owner: Engineering platform team.

Data flow

Slash command lifecycle

  1. Support agent types /sub-status agent@example.com in Slack.
  2. Slack sends a signed POST to the Worker's public URL with the command payload.
  3. Worker validates the Slack signing-secret HMAC; rejects on mismatch with a 401.
  4. Worker queries Stripe's /customers endpoint by email.
  5. If found, Worker fetches the customer's active subscription via /subscriptions.
  6. Worker formats the response as Slack message blocks (plan name, status, renewal date) and POSTs to Slack's response_url.
  7. Worker writes one row to support_audit with agent ID, timestamp, customer ID, action.

External dependencies

  • Stripe API - source of truth for subscription data (api.stripe.com). When down: the slash command returns "Stripe is currently unreachable - try again in a moment" within 3 seconds. No retry; the agent retries manually.
  • Postgres audit database - shared internal Postgres instance hosting the support_audit table (internal RDS). When down: the slash command continues and returns the lookup, but skips the audit write and emits a Worker-level error metric. Better to serve a lookup with a missing audit row than to fail the lookup.
  • Slack Web API - delivery channel for the response message (slack.com/api). When down: the slash command times out; agent retries.

Tech stack

  • Language: TypeScript 5.x
  • Runtime: Cloudflare Workers (V8 isolates)
  • Framework: none (handler is a plain fetch export)
  • Database: Postgres (existing, shared)
  • Key libraries: stripe (npm), @neondatabase/serverless (Postgres driver compatible with Workers)

Deployment topology

  • Cloud account / region: Cloudflare account, deployed globally to all edge locations.
  • Environments: production only (staging is a separate Slack app pointed at the same code via Wrangler env).
  • Ingress: public HTTPS endpoint at https://sub-status.<your-account>.workers.dev; Slack signature verification gates all requests.
  • Data residency: no customer data stored. Stripe data is fetched on-demand and not retained beyond the response. Audit rows live in the existing Postgres instance.

Cross-cutting concerns

Security

Slack signing-secret HMAC verification on every request. Stripe API token lives in Cloudflare Worker secrets, never in code. The Worker has read-only Stripe scope (Customers, Subscriptions). Postgres connection uses a least-privilege role with INSERT-only access to support_audit.

Availability

Cloudflare Workers' platform availability is the practical bound (~99.99%). The tool has no self-managed availability target since it is a low-volume internal lookup; if down, agents fall back to opening Stripe directly.

Observability

Worker logs to Cloudflare's request log (7-day retention). Custom metrics emitted via analyticsEngine for: lookup count, lookup-not-found count, Stripe error count, audit-write failure count. Dashboard in Cloudflare Analytics; no separate observability stack.

Scale

Worker isolates auto-scale; the bottleneck is Stripe's per-account API rate limit (100 read requests/sec). Internal volume is well under 1 req/sec average; no concern.
