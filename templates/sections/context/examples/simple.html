Context

A small internal Slack tool for the Support team to look up customer subscription status without leaving their workflow.

Background

Support agents handle ~40 "what plan is this customer on?" questions per day. Today the answer requires opening Stripe in another tab, searching by email, and copy-pasting the plan name and renewal date back into Slack. Average time to answer: 90 seconds. With a team of 8 agents, that is roughly 8 hours per week of context-switching toll.

The Support lead asked Engineering to build a Slack slash command that returns the same information in under 5 seconds. The ask is small enough that it has been informally floated for 6 months but never prioritised against larger work.

Sponsor and stakeholders

Sponsor

Internal sponsor: Maya Chen, Head of Customer Support. Cares primarily about agent time-on-task and ticket resolution time. Has a quarterly OKR around reducing average handle time.

Other stakeholders

  • Support agents (team of 8) - want the tool to be fast and to never block their existing workflow. Will not adopt anything that adds friction.
  • Devon Park, Engineering Manager - cares about not adding new infrastructure. The tool must run on existing Cloudflare Workers; no new vendors, no new on-call surface.
  • Security team - cares that no customer data is stored in the tool. Audit logging is required for any read of customer subscription state.

Business motivation

Recovering ~8 hours/week of agent time and reducing average handle time on subscription questions from 90 seconds to under 5 seconds. Maya's quarterly OKR target is a 15% drop in average handle time across all ticket types; subscription lookups are the largest single contributor to that average, so the tool credibly contributes 5-7% of the OKR movement on its own.

Adjacent systems and prior art

Existing systems

  • Stripe - source of truth for all subscription data. The tool reads from Stripe via the existing company API token.
  • Slack workspace - the tool ships as a Slack app installed in the existing company workspace.
  • support_audit table - existing audit log used by other internal tooling. The tool writes lookups here, no new infrastructure.

Prior attempts

  • "Subscription Hub" Notion page (2024) - manually-maintained sheet linking Stripe URLs by customer name. Stale within weeks; abandoned. Confirmed the manual approach does not work and the lookup needs to be live.

External constraints

  • No new vendors. Engineering's Q3 mandate is "no new third-party services without committee approval." The tool must use only Stripe and Slack APIs.
  • Two-week build window. The work is squeezed between two larger projects on Engineering's quarterly roadmap.
  • Privacy review. Any new internal tool that accesses customer data triggers a Privacy review (~3 day SLA). Build accordingly.

Definitions

No project-specific terminology. Standard Stripe and Slack vocabulary applies (subscription, customer, plan, slash command, ephemeral message).
