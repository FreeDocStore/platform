Glossary

Terms used throughout the sub-status Slack tool documentation. Most are standard Stripe and Slack vocabulary; a few have project-specific meaning worth pinning down.

agent
A member of the Customer Support team using the sub-status tool. Distinct from "support agent" in Stripe's documentation, which refers to a Stripe-side concept we do not use.
Synonyms: support agent, support team member.
audit row
One entry in the support_audit Postgres table. Every successful sub-status lookup writes exactly one audit row; failed lookups write a row only if the lookup itself succeeded but the audit-write path was the failure point.
See also: audit-write failure.
audit-write failure
The case where a sub-status lookup succeeds but the corresponding row cannot be written to the audit table (typically because the audit DB is briefly unavailable). Per ADR-3, the lookup result is still returned to the agent and a metric is emitted.
See also: audit row.
customer
A Stripe customer record - the entity returned by Stripe's /customers endpoint. In this project, "customer" always refers to the Stripe sense, never to other potential meanings (Salesforce account, internal CRM record).
lookup
One execution of the /sub-status slash command. Each lookup involves: signature verification, one Stripe call, one response to Slack, one audit row.
plan
The product tier a customer is currently subscribed to. Reported to the agent in the slash command response. In Stripe terms, the plan name is read from the customer's active subscription's price.product.
renewal date
The date the customer's current subscription billing period ends and the next period begins. For dunning customers, this is the date Stripe will retry the failed payment, not the original renewal date.
Example: "Renewal date: 2026-05-12" in the slash command response means the next billing event is scheduled for that date.
signing secret
The shared secret used to verify that a slash-command request actually came from Slack. Stored in Cloudflare Worker secrets, never logged. The HMAC verification uses the algorithm Slack documents (v0 signature scheme).
slash command
The Slack mechanism that lets a user trigger a registered handler by typing /sub-status <email> in any channel where the app is installed. The Slack app sends an HTTP POST to the registered URL; the handler returns either an immediate response or invokes the response_url asynchronously.
