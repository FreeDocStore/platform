Requirements

A small internal tool that lets the support team look up customer subscription status without leaving Slack.

Goals

Reduce the time a support agent spends answering "what plan is this customer on?" from ~90 seconds (open Stripe, search, copy plan name) to under 5 seconds (Slack slash command, paste customer email).

Functional requirements

Lookup

  • FR-1. A Slack slash command /sub-status <email> returns the customer's plan name, billing status, and renewal date. (Must)
  • FR-2. If the email matches no customer, the command replies with "No customer found - check the address." (Must)
  • FR-3. Lookup also accepts the Stripe customer ID (cus_*) for cases where the agent already has it. (Should)

Permissions

  • FR-4. The command works only in the #support channel and any channel matching #support-*. (Must)

Non-functional requirements

Performance

  • NFR-1. The slash command responds within Slack's 3-second timeout for 99% of requests, falling back to an async response on slower lookups.

Security

  • NFR-2. The Slack app does not store customer data; each request fetches fresh from Stripe via the company's existing API token (no new credentials issued).
  • NFR-3. Audit log: every lookup writes a row to the existing support_audit table with agent ID, timestamp, and customer ID.

Constraints

  • Budget: Internal tool, no separate budget. Built using existing Stripe + Slack accounts.
  • Timeline: Two-week build window; ship the next sprint.
  • Stack: Must run as a Cloudflare Worker (existing infra). No new vendors.
  • Integrations: Stripe API and Slack API only. No CRM lookup in v1.

Out of scope

  • Updating subscriptions from Slack (read-only tool).
  • Looking up cancelled customers older than 18 months.
  • Multi-account support - the tool reads from the single Stripe production account only.
  • Self-service customer access. This is for support agents only.

Open questions

  • Q-1. Should the response include the customer's CSM/AE? Need to confirm whether that field is consistently populated in Stripe metadata. (Owner: support lead, due: end of week)
  • Q-2. Do we need rate limiting? Stripe API has its own limits; unclear whether the support team's typical volume comes anywhere close. (Owner: engineering, due: before launch)
