Requirements

A regulated-industry case management platform that lets caseworkers intake, track, and close client cases while meeting state-level audit and data-residency obligations.

Goals

Replace the legacy paper-and-spreadsheet workflow with a single web application that (a) reduces case intake time by 50%, (b) produces audit-ready records by default, and (c) gives supervisors real-time visibility into caseload distribution. The platform must onboard the first 200 caseworkers without disrupting in-flight cases.

Functional requirements

Case intake

  • FR-1. Caseworkers create new cases via a guided multi-step intake form covering client demographics, presenting issue, risk assessment, and consent capture. (Must)
  • FR-2. All intake fields support partial save and resume across sessions; no data is lost on browser crash or network drop. (Must)
  • FR-3. Intake includes mandatory risk-assessment fields per the state safeguarding standard; the form blocks submission if risk-tier is unanswered. (Must)
  • FR-4. Caseworkers can attach documents (PDF, image, audio note) up to 50 MB per file; up to 20 files per case in v1. (Should)

Case lifecycle

  • FR-5. Cases progress through five states: Intake, Open, Active, On Hold, Closed. State transitions are explicit user actions, never automatic. (Must)
  • FR-6. Supervisors can reassign cases between caseworkers with a required reason field; both old and new caseworker receive notification. (Must)
  • FR-7. Closing a case requires a closure summary and outcome code from a controlled vocabulary. (Must)

Reporting and audit

  • FR-8. Every record edit writes an immutable audit entry capturing user ID, timestamp, before/after values, and IP. (Must)
  • FR-9. Supervisors can export caseload reports (per worker, per region, per outcome) as PDF or CSV. (Must)
  • FR-10. The platform produces the quarterly state-mandated reporting bundle automatically; supervisors review and submit. (Should)

Non-functional requirements

Performance

  • NFR-1. Page-to-page navigation completes in under 800 ms p95 for caseworkers on a 4G mobile connection.
  • NFR-2. Search across the active caseload returns results in under 1.5 s p95 for queries returning fewer than 200 hits.

Security and compliance

  • NFR-3. All data at rest is encrypted using AES-256; all data in transit uses TLS 1.3 minimum.
  • NFR-4. Authentication is via the agency's SSO (SAML 2.0); MFA is required for any user with supervisor or admin role.
  • NFR-5. Production data is hosted within the state's mandated jurisdiction; no PII leaves that region under any circumstance, including disaster recovery.
  • NFR-6. Penetration test passes annually with no Critical or High findings outstanding more than 30 days.
  • NFR-7. Right-to-erasure requests can be fulfilled within the legally required window; the platform produces a tombstone for any erased record so audit trails remain consistent.

Accessibility

  • NFR-8. Meets WCAG 2.2 AA for all caseworker-facing pages, verified by an external audit before go-live.

Availability

  • NFR-9. Service availability target is 99.9% (excluding declared maintenance windows) measured monthly; planned maintenance occurs only outside business hours in the deployment region.
  • NFR-10. Recovery point objective (RPO): under 15 minutes. Recovery time objective (RTO): under 4 hours.

Constraints

  • Budget: Capped at $X for build phase, $Y/year ongoing operations. Numbers tracked in the Estimates section.
  • Timeline: Pilot with 20 caseworkers in Q3 2026; full rollout (200 users) by end of Q1 2027.
  • Stack: Must run on the agency's existing Azure tenancy (no AWS or GCP). Must use the agency-licensed SSO provider (no separate identity store).
  • Regulatory: Conforms to the state's safeguarding-standard data handling rules; subject to annual audit by the state oversight body.
  • Integration: Must integrate with the existing finance system for billing reconciliation (read-only access via existing API).

Out of scope

  • Mobile native apps. Web-first only; the responsive web app must work on caseworker tablets.
  • Inter-agency case sharing. Cases are scoped to the deploying agency in v1; cross-agency referrals are a v2 capability.
  • Client-facing portal. Only caseworkers, supervisors, and admins use the platform in v1; client self-service is a separate program.
  • Migration of cases closed before 2024. Historical archival lives in the legacy system; v1 carries forward only active and recently-closed cases.
  • AI-driven case triage. The risk assessment is rule-based and deterministic; no ML-driven prioritisation in v1.

Open questions

  • Q-1. What is the legally required maximum window for fulfilling right-to-erasure (NFR-7)? Need confirmation from the agency's legal team. (Owner: project sponsor, due: before architecture sign-off)
  • Q-2. Are the document attachment size and count limits (FR-4) sufficient, or do we need to handle larger evidence packs? Need 5 caseworker interviews. (Owner: product lead, due: end of discovery)
  • Q-3. Does the existing SSO provider support the MFA requirements in NFR-4 without additional licensing? (Owner: security architect, due: end of week)
  • Q-4. The state's data-residency boundary: is it state-line or country-line? This decides which Azure regions are eligible. (Owner: legal + sponsor, due: before infrastructure design)
